상품(카탈로그)별 네이버 여론을 모아 출처·인용에 접지된 인사이트를 뽑는다. 동기 경로 대비 비용 절반, 대신 비동기(최대 24시간). 크롤·LLM·적재를 세 단계로 분리해 언제든 재개할 수 있다.

01 한눈에

SKU 하나마다 세 번의 구조화 추출 콜을 던진다.

추출 코어는 순수 패키지 insight_engine/에 있고, 실제 배치를 돌리는 오케스트레이터는 db/run_insight_batch_openai.py에 있다. 기존 naver_review_geo의 수집·인용검증 로직을 그대로 재사용하므로, 배치로 뽑은 인사이트도 동기 경로와 품질이 동일하다. 모든 결과에는 어떤 엔진·프롬프트·모델 버전으로 나왔는지 run_meta가 박힌다.

03 3단계 파이프라인

제출과 회수 사이는 최대 24시간 — 상태는 로컬 manifest·staging에 남아 세션·재시작을 넘어 이어진다.

submit

제출

  1. 미처리 SKU 큐 구성 (insight 없는 카탈로그)
  2. SKU별 네이버 블로그 크롤 → items
  3. staging.jsonl에 items 저장
  4. SKU×3 요청 .jsonl 빌드 → 청킹
  5. OpenAI 파일 업로드 → 배치 생성
→ manifest.json (batch_ids)
OpenAI · 최대 24h

status

대기·조회

  1. 배치별 상태 폴링
    (in_progress → completed)
  2. 완료 카운트 확인
  3. 비용·크롤 없음 — 무료
← batch status

fetch

회수·적재

  1. 완료 배치 output 다운로드
  2. custom_id로 SKU·스키마 되매핑
  3. staging items로 인용검증·재접지
  4. run_meta 스탬핑 → Mongo 적재
  5. 이미 있으면 skip (멱등)
→ products.catalogs[].insight

05 환경 설정 (env)

키는 run.sh 한 파일에 모으고, 스크립트가 자동으로 읽어들인다. 직접 export 하거나 한 줄 오버라이드도 가능하다.

필요한 변수

변수필수기본값설명
NAVER_CLIENT_ID✔—네이버 검색 API 클라이언트 ID (블로그 크롤)
NAVER_CLIENT_SECRET✔—네이버 검색 API 시크릿
OPENAI_API_KEY✔—OpenAI 키 (Batch 제출·회수)
INSIGHTS_DB—insights_demo대상 DB (래퍼 기본값. 운영은 insights)
MONGO_URI—localhost:47017Mongo 접속 (directConnection)
INSIGHT_MODEL—gpt-4o-mini추출 LLM 모델
run.sh 작성 — 템플릿 복사 후 키만 채우기
# insight/ 에서. 예제 템플릿을 복사
cp run.sh.example run.sh
# run.sh 를 열어 <YOUR_VALUE_HERE> 를 실제 키로 채운다
export NAVER_CLIENT_ID="실제_ID"
export NAVER_CLIENT_SECRET="실제_시크릿"
export OPENAI_API_KEY="sk-..."
적용 — 래퍼가 자동 로드 (직접 할 필요 없음)
# run_batch_insight.sh 는 실행 시 run.sh 의 export 를 스스로 읽는다:
#   set -a; eval "$(grep '^export ' run.sh)"; set +a
# 그래서 그냥 아래처럼 바로 실행하면 된다
db/run_batch_insight.sh submit

# 직접 셸에 로드하고 싶다면(수동 실행·디버깅용):
set -a; eval "$(grep '^export ' run.sh)"; set +a
오버라이드 — 한 번만 다른 DB/모델로
# 앞에 변수만 붙이면 그 실행에만 적용된다
INSIGHTS_DB=insights db/run_batch_insight.sh fetch
INSIGHT_MODEL=gpt-4o db/run_batch_insight.sh submit 500
🔐
run.sh 는 실키를 담는다 — 절대 커밋 금지. 이미 .gitignore에 있다. 공개 미러 repo에도 키·데이터가 들어가지 않도록 순수 패키지만 동기화한다.

06 실행 방법

환경이 준비되면 래퍼 db/run_batch_insight.sh 하나로 세 단계를 돌린다. (작업 디렉토리 = insight/)

선행 준비 — Mongo 기동 (env는 §05에서 완료)
# MongoDB(pig-mongo) 기동
open -a Docker && docker start pig-mongo
# 키는 §05 run.sh 에 있고 래퍼가 자동 로드 — 추가 작업 없음
제출 — 오늘 하루치 (네이버 일일쿼터 상한)
# 기본 25,000 SKU · 대상 INSIGHTS_DB=insights_demo · 백그라운드 크롤
db/run_batch_insight.sh submit
# 규모 조절: 검증용 소량
db/run_batch_insight.sh submit 500
# 진행 확인(크롤 건수)
wc -l db/insight_engine_batch/run_*/staging.jsonl
상태 조회 — 배치 완료 대기 (무료)
# 제출한 배치들의 status (수분~24h 후 completed)
db/run_batch_insight.sh status
회수·적재 — Mongo 반영 (멱등, 완료까지 반복)
# 완료된 배치만 회수해 catalogs[].insight 에 적재
db/run_batch_insight.sh fetch
# 결과: {"loaded": N, "skipped": M, "pending_batches": K}
# pending_batches 가 0 될 때까지 반복 호출
⚠️
대상 DB 주의. 대량 카탈로그는 insights_demo에 있고 운영 insights는 현재 미처리 0건이다. 운영 반영이 필요하면 INSIGHTS_DB=insights db/run_batch_insight.sh …로 명시한다. submit은 실행 전 대상 DB·건수·예상 비용을 출력한다.

10 인수인계 노트

이어받는 사람이 알아야 할 재개·문제해결·한계·검증.

재개 — 어디서 멈춰도 이어진다

상태는 전부 db/insight_engine_batch/run_*/에 남는다 — staging.jsonl(크롤한 items)과 manifest.json(batch_id들). 세션·터미널을 닫아도 무관.

  • submit 중단 → 다시 submit. 이미 insight 있는 SKU는 큐에서 자동 제외.
  • fetch는 멱등 → 완료될 때까지 몇 번을 호출해도 이미 적재된 건 skip. pending_batches가 0 될 때까지 반복.
  • 특정 run 재개 → BATCH_RUNDIR=db/insight_engine_batch/run_… db/run_batch_insight.sh fetch

트러블슈팅

증상원인 · 해결
Mongo Connection refusedpig-mongo 미기동 → docker start pig-mongo (Docker 데몬 먼저)
크롤이 중간에 느려지거나 멈춤네이버 일일 쿼터(~25k) 소진 → 남은 SKU는 skip. 다음 날 다시 submit이 이어받음(정상)
manifest.json 안 생김크롤 진행 중이거나 실패 → 로그 db/batch_openai_submit_*.log 확인
fetch 결과 loaded:0배치 아직 미완(status로 확인) 또는 이미 적재됨(skipped 확인)
제출 도중 에러로 죽음이미 생성된 배치는 OpenAI 대시보드에서 조회·취소 가능(아래 한계 참고)

알려진 한계 (fast-follow)

  • 청크 제출 루프 중간 실패 시 이미 만든 배치가 manifest에 안 남을 수 있다 → OpenAI 대시보드에서 수동 확인. (증분 manifest로 개선 예정)
  • fetch는 호출마다 output을 다시 다운로드한다 — 대규모에서만 비효율.
  • 수집(크롤)이 네이버 쿼터에 묶여 전체 소진은 ~16일. 일 단위로 나눠 돌린다.

11 실행 모드 — sync ↔ batch 혼용

배치 하나만이 아니다. 같은 API에서 execution으로 실시간(sync)과 배치(batch)를 골라 혼용한다.

모드지연비용용도
sync (기본)즉시정가실시간 단건·소량
batch최대 24h50%↓대량 백필

router.submit

단일 진입점

  1. cfg.execution으로 분기
  2. 급한 단건·소량 → sync
  3. 대량·비용민감 → batch
→ 호출자가 모드만 지정

sync

실시간

  1. 인프로세스 잡 큐 드레인
  2. 즉시 결과 반환
  3. Mongo 불필요(파일 상태)
→ {mode, job_id, state}
50% ↓

batch

비동기

  1. 크롤 → 요청빌드 → OpenAI 제출
  2. batch_openai lazy import
  3. staging 반환(호출자 영속화)
→ {mode, batch_ids, staging}
REST — execution 으로 모드 선택
# 실시간 단건
curl -XPOST :8770/extract -d '{"keyword":"아식스 젤카야노"}'

# 잡 — sync(즉시) vs batch(저렴·비동기)
curl -XPOST :8770/jobs -d '{"execution":"sync",  "targets":[{"keyword":"a","uid":"C1"}]}'
curl -XPOST :8770/jobs -d '{"execution":"batch", "targets":[{"keyword":"a","uid":"C1"}]}'

# 배치 진행 조회 · 관측
curl -XPOST :8770/batch/status -d '{"batch_ids":["batch_..."]}'
curl :8770/metrics
라이브러리 — router.submit
from insight_engine import router
from insight_engine.types import ExtractTarget, EngineConfig

r = router.submit(targets, EngineConfig(execution="sync"))    # 즉시
r = router.submit(targets, EngineConfig(execution="batch"),  # 50%↓
                  client=client, creds={"nid":..,"nsec":..})
🧩
sync-only 배포는 가볍다. batch_openai는 batch 경로에서만 lazy import → 실시간 REST만 쓰면 Mongo·배치 의존 없이 openai·pydantic·requests만으로 돈다.
